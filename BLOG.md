# Synthetic NHS Patient Journeys, One Serverless GPU Job at a Time

*Built for the Nebius hackathon*

## The problem

Healthcare ML and clinical NLP teams are stuck in a bind: they need
realistic patient data — demographics, admission records, clinical
notes written the way a doctor actually writes them — to build and test
pipelines, but real patient data is (rightly) locked down. Existing
synthetic-data tools tend to fall into two traps. Either they hardcode
a single coding standard (good luck if your team works in SNOMED CT
instead of ICD-10), or they generate plausible-looking text without
checking whether it actually reflects the codes you asked for — you end
up with a "STEMI" diagnosis code attached to a discharge summary that
never mentions a heart attack.

**serverless-journeys** generates full synthetic NHS patient journeys —
patient, admission, event timeline, and clinical notes — from
diagnostic and procedure codes, and it runs as a single serverless GPU
job on Nebius: no standing inference endpoint, no separate model-serving
infrastructure. One container, one job, a batch of patients out the
other end.

## What it does

Point it at a diagnostic code — say `I21.0`, a STEMI (heart attack) —
and it produces:

- **A patient**: NHS-style demographics, allergies, past medical
  history, medications, social history.
- **An admission**: specialty, ward, chief complaint, working
  diagnosis, management plan, length of stay — all consistent with the
  driving code.
- **A journey**: an ordered clinical timeline (ED review, post-take
  ward round, operation, therapy review, ...), sized to a configurable
  target.
- **Clinical notes**: one per journey event, written in the actual
  style clinicians use for that note type — SBAR/NEWS2 bullet points
  for an ED note, ASGBI-standard prose for an operation note, formal
  ward-round documentation for a consultant review.

Every one of those documents is generated by a real LLM call, prompted
with the clinical context of the codes you gave it — not templated text
with the diagnosis name swapped in.

### It checks its own work

The part that matters most for data quality: after generation, a
**backward pass** checks that every driving code you asked for is
actually reflected in the admission record or the notes — not just
stamped on as metadata. If a code didn't make it into the narrative
(the LLM drifted, or generated something generic), the pipeline makes a
**targeted corrective call** — rewriting just the admission's narrative
fields, or just one note — and re-checks. No full patient regeneration,
no silent failures: codes that still don't land after the configured
number of attempts are logged and reported in the run summary, so you
always know exactly what you got.

### Any coding standard, no code changes

Diagnostic and procedure codes are pluggable. ICD-10 and OPCS-4 ship
built-in, but adding ICD-11, SNOMED CT, CPT, or a private classification
is a matter of dropping a JSON file into `code_systems/` — the pipeline
code has no ICD-10/OPCS-4-specific logic baked in. For codes with no
curated entry in the dictionary, an optional research mode looks the
code up via Google Custom Search first, so generation is grounded in
what the code actually means rather than the LLM guessing from the bare
code string.

### Runs as a single Nebius GPU job

This is the part built specifically for Nebius: `Dockerfile.gpu`
packages a vLLM server *and* the generation pipeline into one image.
On startup, the container launches vLLM locally, waits for the model to
finish loading onto the GPU, points the pipeline at
`http://127.0.0.1:8000/v1` (Nebius's serverless endpoints and vLLM's
server are both OpenAI-wire-compatible, so the same client code just
works), runs the full batch, and shuts vLLM down cleanly on exit. Launch
one Nebius GPU job — `--platform gpu-l40s-a` for a 7-8B model, a
multi-GPU/H100 preset for 70B-class — and you get model serving and
generation in a single ephemeral unit, with nothing left running
afterward. A second, GPU-free image (`Dockerfile`) exists for the case
where you'd rather call an already-running Nebius AI Studio endpoint
(or Anthropic/OpenAI) instead of self-hosting the model.

## Technical breakdown

### Pipeline architecture

`main.py` runs a fixed, linear pipeline per patient
(`run_pipeline`): parse codes → generate patient → generate admission →
generate journey → generate notes → verify/correct → (optionally)
evaluate → save. Each stage is one or more LLM calls through a single
dispatch function, `processing.call_llm(prompt, model, ...)`, which
routes to `_call_anthropic`, `_call_openai`, or `_call_nebius` based on
`LLM_PROVIDER`. There's no abstract provider interface — the interface
*is* the shared function signature. Anthropic uses
`messages.create(...)`; OpenAI and Nebius both use
`chat.completions.create(...)` via the same `openai` SDK, since
Nebius's endpoint is OpenAI-wire-compatible — Nebius support is
`_get_nebius_client()` pointing `openai.OpenAI(base_url=...)` at
`NEBIUS_BASE_URL` instead of OpenAI's default. All three paths retry
with exponential backoff and raise after `MAX_LLM_ATTEMPTS`.

A patient that throws at any stage is logged and skipped — the batch
keeps going rather than aborting the whole run.

### The code registry: data, not code

`src/codes/registry.py` defines a generic `CodeSystem` dataclass (`key`,
`name`, `kind`, `codes`, `specialty_field`, `type_field`, `chapter_map`,
`default_specialty`) and a module-level registry with
`register_code_system()` / `get_code_system()`. Every function that
touches codes — `parse_codes`, `lookup_code`, `infer_specialty`,
`get_clinical_context` — operates on a `CodeSystem` value, never on a
hardcoded standard. `src/codes/loader.py` discovers and validates JSON
files under `code_systems/` (plus an optional
`$EXTRA_CODE_SYSTEMS_DIR`, designed for mounting a private code-system
volume into a Nebius job without an image rebuild) and registers them
on import. A malformed file logs a warning and is skipped rather than
crashing the run. `tests/test_code_registry.py` proves the genericity
by registering a third, entirely made-up code system and running it
through the same code paths as ICD-10/OPCS-4.

Worth noting: `code_systems/opcs4.json` currently ships with an empty
`codes` dict — an NHS classbrowser audit (commit `7c6ddaa`) found wrong
specialty mappings in the curated OPCS-4 data and it was pulled pending
re-curation, rather than shipping known-bad data. Every OPCS-4 code
currently falls through the generic/researched path. That's a real
example of the data-quality auditing this project does on itself.

### The backward pass, in detail

`processing.check_code_reflected(code, code_system, patient, admission,
journey, notes)` builds a "needles" set — the code itself plus
significant words (4+ letters) from its curated description — and
substring-searches it against four JSON-serialized artifacts. The
admission check deliberately excludes bookkeeping fields
(`diagnostic_codes`, `code_reflection_check` itself) via
`_ADMISSION_CODE_BOOKKEEPING_FIELDS`, so a code can't trivially "pass"
just because it's echoed back as metadata — it has to appear in the
actual narrative.

The correction loop, in `main.step_verify_code_reflection`:

```python
result = processing.check_code_reflected(code, code_system, patient, admission, journey, notes)
attempts = 0
while not (result["admission"] or result["notes"]) and attempts < max_correction_attempts:
    attempts += 1
    code_context = processing.get_code_context(code_system, code, enable_research=..., model=model)
    admission.update(processing.correct_admission_for_code(admission, code, code_context, model=model))
    target_note = _select_note_for_correction(notes, role)  # operation note for procedures, else earliest
    if target_note is not None:
        corrected_text = processing.correct_note_for_code(target_note["clean_note_text"], code, code_context, model=model)
        target_note["clean_note_text"] = corrected_text
        target_note["raw_blob_content"] = corrected_text
    result = processing.check_code_reflected(code, code_system, patient, admission, journey, notes)
report[f"{role}:{code}"] = result
```

Each correction is a single, targeted LLM call — rewrite the admission's
narrative fields, or rewrite one note — never a full patient re-run.
Results are keyed `role:code` (not just `code`) specifically because
the same code string can appear as both a diagnostic and a procedure
code across different systems, and an earlier version of this collided
on that (fixed in commit `70b2333`, along with a single-note correction
bias where every correction landed on the same note regardless of
code role). Codes still unreflected after `max_correction_attempts`
(default 1; `0` disables correction entirely) are logged as warnings and
surfaced in `generation_summary.json`'s `code_reflection_check.
unreflected_codes` — the run tells you exactly what didn't land instead
of silently shipping it.

### Two Docker images, one purpose split

`Dockerfile` is `python:3.12-slim` plus the pipeline — for calling an
already-running endpoint (Anthropic, OpenAI, or Nebius AI Studio), no
GPU needed. `Dockerfile.gpu` is built on `vllm/vllm-openai:latest`,
adds the pipeline on top, and swaps the entrypoint for `entrypoint.sh`:

```bash
python3 -m vllm.entrypoints.openai.api_server \
    --model "${MODEL}" --port "${VLLM_PORT}" \
    --gpu-memory-utilization "${VLLM_GPU_MEM_UTIL}" ${VLLM_EXTRA_ARGS:-} &
# poll /v1/models until ready or VLLM_STARTUP_TIMEOUT elapses,
# trap EXIT to kill vLLM on the way out
export LLM_PROVIDER=nebius
export NEBIUS_BASE_URL="http://127.0.0.1:${VLLM_PORT}/v1"
python3 main.py "$@"
```

Because the generation pipeline's Nebius provider is just the OpenAI
SDK against a configurable `base_url`, pointing it at a local vLLM
server instead of Nebius's hosted endpoint required zero pipeline code
changes — only the entrypoint script changed. That's the same
OpenAI-wire-compatibility that makes Nebius AI Studio itself a drop-in
provider in the first place. `tests/test_fake_endpoint.py` exercises
this at the wire level: it spins up a real local HTTP server speaking
the OpenAI chat-completions format and points the Nebius provider at
it, so the Nebius integration is tested without needing real
credentials in CI.

CI (`.github/workflows/docker-publish.yml`) builds and pushes both
images to GHCR on every push to `main`, tagged `:latest`/`:latest-gpu`
plus sha-tags, using a build matrix over the two Dockerfiles.

### Output

Four CSVs (`synthetic_patients`, `synthetic_admissions`,
`synthetic_journeys`, `synthetic_clinical_notes`) plus a
`generation_summary.json` with run stats, the code-reflection report,
and (with `--evaluate-notes`) readability and LLM-judged quality scores
per note. No output schema is hardcoded via a pydantic model — the LLM's
JSON response is the schema, enriched with bookkeeping IDs and
serialized with pandas — which keeps the pipeline flexible as prompts
evolve, at the cost of the output shape being defined by the prompts
rather than a strict contract. That trade-off felt right for a
synthetic-data generator whose whole value proposition is "as
realistic and varied as the underlying LLM can produce."

### Testing philosophy

13 test files cover forward-pass (codes actually reach the prompts),
backward-pass (the reflection check itself), full pipeline
correction/reflection wiring end-to-end with `call_llm` mocked, registry
genericity, code research/search clients, quality metrics, and the
Nebius wire format against a fake local endpoint. A `--test-mode` flag
bypasses all LLM calls with stub data for cheap pipeline smoke-testing.
Nothing here needs real API credentials to validate — which matters for
a project whose core value is a live LLM integration.

## What's next

The most honest thing in this codebase is what's *missing*: OPCS-4
curated data was pulled after an audit found it wrong, rather than
shipped anyway. Re-curating that data against the NHS classbrowser is
the immediate next step, followed by adding SNOMED CT as a second
built-in coding standard to prove the pluggable registry design beyond
ICD-10/OPCS-4 in practice, not just in tests.
